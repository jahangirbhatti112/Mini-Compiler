# C++ Mini-Compiler: Phase 1 & Phase 2 Documentation

This section provides an in-depth breakdown of the **Frontend Components** of the compiler pipeline, which are responsible for analyzing the source code structure, validating grammar, and building the initial in-memory representation.

---

## 🎯 Phase 1: Lexical Analysis (Scanning)

The Lexical Analyzer (Lexer) acts as the compiler's reader. Its primary responsibility is to scan the raw input character stream from left to right, strip away whitespaces, and group valid characters into meaningful atomic units called **Tokens**.

### Core Component Files
* **`src/Token.h`**: Declares the `TokenType` enumeration (`TOKEN_KEYWORD`, `TOKEN_IDENTIFIER`, `TOKEN_NUMBER`, `TOKEN_OPERATOR`, `TOKEN_SEMI`, etc.) and the core `Token` data structure that pairs a type tag with its literal text value.
* **`src/Lexer.h`**: Implements the main state-machine scanner loop. It utilizes pointer-advancing lookahead utilities (`peek()`, `advance()`) to categorically extract variables, numeric constants, and reserved system keywords (`int`, `while`).

---

## 🌿 Phase 2: Syntactic & Semantic Analysis (Parsing)

The Syntactic and Semantic Analyzer (Parser) acts as the compiler's structural judge. It ingests the flat sequence of tokens generated by Phase 1, verifies that the syntax adheres to the strict context-free grammar rules of our custom language subset, and handles type and declaration binding.

### Core Component Files
* **`src/AST.h` (Abstract Syntax Tree)**: Defines the base `ASTNode` interface and its derived concrete class configurations (`AssignmentNode`, `WhileNode`, `BinaryOpNode`, `BlockNode`). These node variations are dynamically engineered to build an hierarchical blueprint tree of the code in memory.
* **`src/Parser.h`**: Implements a recursive-descent parsing strategy. It features lookahead matching functions (`match()`) and recursive control-flow logic (`parseBlockOrStatement()`) to structurally handle scoped blocks encased inside curly brackets `{ }`.

### ⚡ Integrated Compile-Time Optimization (Constant Folding)
Embedded directly inside **`src/Parser.h`** is a high-efficiency performance phase known as **Constant Folding**. 

When the parser encounters a mathematical operations node composed entirely of literal numbers (e.g., `5 + 2`), it intercepts the tree generation process. Instead of offloading separate runtime additions onto the CPU backend, the parser pre-calculates the arithmetic expression at compile-time and drops a flattened, pre-optimized single element directly into the pipeline:
